1. Arsitektur Sistem

Sistem dirancang menggunakan arsitektur modern berbasis Client-Server (Decoupled). Hal ini memisahkan layanan menjadi dua entitas mandiri yang bisa di-scale secara terpisah

  • Weather API: Open-Meteo (Surabaya Coordinates: -7.25, 112.75)
  • Frontend: React.js difungsikan sebagai Single Page Application (SPA) yang optimal untuk sisi PWA (Progressive Web App)
  • Backend: Node.js / Express.js melayani business logic sebagai REST API yang asinkron
  • Database Utama: PostgreSQL dengan sistem ORM (Prisma) untuk menjamin relasi tabel ACID secara ketat
  • Cache Layer (Bila Perlu): Redis diposisikan di depan layer database untuk menampung query yang paling sering dipanggil ulang oleh client (misal: Konfigurasi App & Rate Limiter) sehingga beban IOPS Database lebih efisien

Cara Membaca Garis Relasi Tabel (Kardinalitas):

  • ||--o{ : Relasi One-to-Many (Zero or More). Contoh: Satu entitas Owner bisa memiliki nol hingga banyak Product
  • ||--|{ : Relasi One-to-Many (One or More). Contoh: Sebuah Lorong rak di toko harus memiliki setidaknya satu Rak penyimpanan
  • PK : Primary Key (ID Utama/Unik)
  • FK : Foreign Key (ID Kaitan dari tabel lain)

2. Akses Super Admin (Level Tertinggi)

Akses akun level SUPER_ADMIN dilindungi secara berlapis pada area controller dan middleware. Hal ini menghalau akses Bypass dari user biasa. Selain itu, admin level ini memiliki akses untuk mem-backup database langsung dari sistem

Implementasi Middleware Validasi Role

backend/src/middlewares/auth.middleware.js
const verifySuperAdmin = (req, res, next) => {
  // Pastikan token sudah di-decode pada proses sebelumnya
  if (!req.user) {
    return res.status(401).json({ success: false, message: "Akses ditolak. Token tidak valid." });
  }

  // Cek otorisasi level tertinggi
  if (req.user.role !== 'SUPER_ADMIN' && req.user.role !== 'OWNER') {
    return res.status(403).json({ 
      success: false, 
      message: "Forbidden: Membutuhkan privilege Super Admin." 
    });
  }

  // Log aktivitas sensitif (Audit Trail)
  console.log(`[AUDIT] SuperAdmin ${req.user.email} mengakses route ${req.method} ${req.originalUrl}`);
  next();
};

module.exports = { verifySuperAdmin };

Fungsi Backup Database (Download .sql)

Super Admin dapat men-download dump SQL database secara real-time via endpoint /api/admin/backup.

backend/src/controllers/admin.controller.js
const { exec } = require('child_process');
const path = require('path');
const fs = require('fs');

const downloadBackup = (req, res) => {
    // Definisi nama file dan path sementara
    const dateStr = new Date().toISOString().replace(/[:.]/g, '-');
    const fileName = `aichat_backup_${dateStr}.sql`;
    const tempPath = path.join(__dirname, '../../tmp', fileName);
    
    // Command pg_dump PostgreSQL
    // env PGPASSWORD diset di level child_process agar tidak terbaca dari ps aux
    const dumpCmd = `pg_dump -U ${process.env.DB_USER} -h ${process.env.DB_HOST || 'localhost'} -d ${process.env.DB_NAME} -f ${tempPath}`;
    
    // Eksekusi terminal server secara child process
    exec(dumpCmd, { env: { ...process.env, PGPASSWORD: process.env.DB_PASS } }, (error, stdout, stderr) => {
        if (error) {
            console.error(`Backup error: ${error.message}`);
            return res.status(500).json({ success: false, message: "Gagal membuat file backup." });
        }
        
        // Kirim stream file ke browser klien
        res.download(tempPath, fileName, (err) => {
            if (err) console.error("Gagal mendownload file:", err);
            // Hapus file sementara setelah klien selesai mengunduh
            fs.unlinkSync(tempPath);
        });
    });
};

module.exports = { downloadBackup };

3. Panduan Backup (Pencadangan Databaase & Server Frontend)

Pencadangan diutamakan berjalan otomatis di Linux Shell (Bash) menggunakan fitur Job Scheduler cron saat VPS server sedang minim aktivitas agar performa tetap 100% saat Peak Hours. Namun, developer juga dapat mem-backup dari Local PC.

A. Lingkungan VPS Server (Production)

Simpan kode ini di VPS server sebagai file backup_sys.sh, lalu daftarkan di crontab linux (crontab -e) untuk berjalan jam 2 pagi. Apabila memiliki storage server cadangan cloud, integrasikan dengan package AWS CLI S3

/scripts/backup_sys.sh
#!/bin/bash
# Konfigurasi Database Lokal PostgreSQL
DB_USER="admin"
# Password di-inject ke env khusus shell 
export PGPASSWORD="password_super_kuat"
DB_NAME="aichat_production"
FRONTEND_DIR="/var/www/aichat-frontend" # Asset Webroot Nginx React PWA

BACKUP_DIR="/var/backups/aichat"
DATE=$(date +"%Y-%m-%d_%H%M")
DB_FILE="$BACKUP_DIR/$DB_NAME-$DATE.sql"
SYS_ARCHIVE="$BACKUP_DIR/full_backup_$DATE.tar.gz"

# Buat folder sementara jika tidak ada di Node Server
mkdir -p $BACKUP_DIR

# 1. Eksekusi proses PostgreSQL pg_dump murni
echo "Memulai ekstraksi tabel dari db PostgreSQL $DB_NAME..."
pg_dump -U $DB_USER -h localhost -d $DB_NAME -f $DB_FILE

# 2. Kompresi file Database SQL & Source Static Web Nginx (Memotong usage disk hingga ~80%)
echo "Mengarsipkan Database dan Build Frontend Nginx..."
tar -czvf $SYS_ARCHIVE $DB_FILE $FRONTEND_DIR
rm $DB_FILE # hapus file .sql yang berat

# Clear memory env
unset PGPASSWORD

# Hapus Log yang umurnya kelewat lama demi efisiensi storage (Misal setelah 30 Hari)
find $BACKUP_DIR -type f -name "*.tar.gz" -mtime +30 -exec rm {} \;

# [OPSIONAL] Push file arsip ke Cloud Object Storage yang Aman & Murah (E.g Amazon S3)
aws s3 cp $SYS_ARCHIVE s3://aichat-backup-bucket/full-backups/

echo "Backup lokal dan cloud S3 sukses: $SYS_ARCHIVE"

6. Maintenance & Trouble Sederhana

Panduan komprehensif penanganan masalah (Troubleshooting) operasional sehari-hari, difilter berdasarkan lingkup eksekusinya:

Perbaikan Khusus Frontend (Client-Side)

1. UI Tidak Berubah Setelah Re-Deploy (Stale Cache)

Penyebab: Browser pengguna masih menyimpan (cache) file Javascript/CSS versi lama yang dikunci oleh Service Worker atau setting HTTP Cache

Solusi IT:

  • Instruksikan pengguna untuk melakukan *Hard Reload* (Ctrl+F5 atau Cmd+Shift+R).
  • Pada saat fase build aplikasi React, pastikan konfigurasi *bundler* (Vite/Webpack) selalu menambahkan hash unik pada nama file fisik (contoh: main.1a2b3c.js).

2. Error Tampilan "Blank White Screen" (React Crash)

Penyebab: Terjadi *Unhandled Exception* pada rendering komponen React (misal, mencoba me-render map/array dari variabel yang ternyata berstatus `undefined` atau null dari API)

Solusi IT:

  • Buka F12 > Console tab, akan terlihat garis merah eksak baris yang bermasalah
  • Bungkus seluruh komponen aplikasi root dengan <ErrorBoundary> untuk menangkap fatal crash agar tidak blank dan malah merender UI "Opps ada masalah"
  • Gunakan *Optional Chaining* (misal: data?.user?.name) ketimbang direct render (data.user.name)

3. CORS Error di Console Browser

Penyebab: Frontend di-serve dari domain-a.com namun mencoba memanggil Endpoint API di api-domain-b.com, dan Backend tidak mengijinkan origin asal Frontend

Solusi IT: Akses ke Backend dilarang sepihak oleh mitigasi browser asli. Solusinya, beritahukan tim backend untuk memasukkan domain frontend persis (beserta https) ke dalam daftar whitelisting Cors options di server Node.js

Perbaikan Khusus Backend (API & Logic)

1. Alur Investigasi Letak Modul (Panduan Tracing Kode)

Ketika menemukan laporan Frontend memunculkan error API, ikuti alur ini untuk melacak file mana di Backend yang bermasalah:

  1. Network Tab (Browser) -> Tekan F12, masuk tab "Network". Temukan baris request URL API yang berhuruf merah (Gagal). Catat URL path akhirnya (Contoh: /api/v1/users/register) dan tipe methodnya (POST).
  2. Lacak di Route Layer -> Buka project Backend, cari pendaftaran URL di folder routes/. (Contoh: Buka file backend/src/routes/user.route.js lalu cari router.post('/register', ...)). Di sana akan tertulis pemanggilan Controller-nya (Misal: authController.registerUser).
  3. Masuk ke Controller -> Temukan dan buka file backend/src/controllers/auth.controller.js, cari fungsi registerUser. Controller bertugas menerima request, namun proses aslinya dilempar ke layer Service.
  4. Isolasi di Service Layer (Business Logic) -> Temukan dan buka file backend/src/services/auth.service.js. Hampir 90% letak kalkulasi rumit, database action, dan error logika bersarang di mari. Disinilah IT memperbaiki masalah sesungguhnya.

2. Panduan HTTP Error Umum (401, 403, 404, 422)

  • Error 401 / 403 (Unauthorized / Forbidden):
    Token JWT kadaluarsa, tidak terbawa di Header, atau akun tidak punya Role yang sesuai. Cek interceptor frontend apakah Authorization: Bearer selalu disisipkan.
  • Error 404 (Not Found):
    URL endpoint gagal dipanggil case-sensitive-nya salah, tidak terdaftar di router backend, atau method-nya beda (Panggil POST tapi di router GET).
  • Error 422 (Unprocessable Entity):
    Client Frontend mengirim muatan JSON (POST/PUT) yang tidak sesuai dengan skema validasi backend (Misal, input tipe email tapi isinya hanya teks biasa). Revisi pengiriman form di komponen Frontend.

3. Request Timeout / Response Ngelag Parah (Api Lambat)

Penyebab: Adanya Database N+1 Query Problem (Kueri looping database yang melahap resourse), atau third-party layanan SMS/Mail yang sedang lambat tapi tidak dibungkus Asynchronous Queue

Solusi IT: Cek file Controller yang memicu lamanya loading. Gunakan fitur pagination untuk fetching data dari database jika listnya ribuan baris, atau buat tabel khusus indexing pada field database bersangkutan

Perbaikan Khusus Server / Infrastruktur

1. Reload Halaman SPA Berujung 404 (Not Found)

Penyebab: URL sekunder (contoh: /dashboard) dikenali React saat berjalan, tapi tidak dikenali Nginx saat direfresh paksa karena file fisik dashboard.html memang tidak pernah ada.

Solusi IT: Tambahkan skrip fallback routing ke index.html pada block location Nginx production

/etc/nginx/sites-available/aichat
server {
    listen 80;
    server_name aichat.domain.com;
    root /var/www/aichat/frontend/build;
    index index.html;

    location / {
        # Fallback semua request rekursif kembali utuh ke index.html
        try_files $uri $uri/ /index.html;
    }
}

2. Error 500 (Internal Server Error) Berkepanjangan

Penyebab: Bug koding syntax (typo error), koneksi Database ditolak, variabel env tidak di load dengan proporsional ke sistem server, atau terjadi memori leak (OOM)

Solusi IT: Pantau error stack-trace spesifik melalui proses manajer pm2 (Node.js daemon):

Deteksi Error PM2 Terminal
# Memantau realtime Live Log yang melempar code 500
pm2 logs aichat-backend

# Me-restart server API bila beban macet 
pm2 reload aichat-backend

3. Gagal Deployment Prisma Migrate di Server

Penyebab: Prisma gagal menyinkronkan data schema ke database produksi saat ada perintah hapus kolom tapi masih ada baris yang bernaung disana, menyebabkan migration conflict.

Solusi IT: Jangan menggunakan prisma migrate dev di server, paksakan eksekusi manual via bash:
Gunakan sinkronisasi produksi: npx prisma migrate deploy
Lalu wajib disusul agar file generated ORM mendeteksi: npx prisma generate.

7. Standar Implementasi API Layer

Penyeragaman interaksi HTTP secara profesional sangat penting agar integrasi pihak frontend dan App Mobile mulus tanpa membongkar logic per route, sekaligus menjaga kestabilan network

Pagination & Metadata Query (Proteksi Kelebihan Beban)

Jangan pernah mengeksekusi fetch "Ambil Semua Transaksi/Tabel" karena akan memicu lag atau Node.js memakan ratusan MB RAM saat tabel membesar. Rest API Enterprise memaksakan Pagination sebagai bentuk paginasi respon, dibatasi per halaman 10 hingga 50 limit data, di-fetch sesuai parameter URL ?page=X&limit=Y.

Contoh Konsep Prisma Pagination Limit di Controller
const getAllUsers = async (req, res) => {
  // Ambil parameter paginasi langsung dari URL, default page 1, limit 20
  const page = parseInt(req.query.page) || 1;
  const limit = parseInt(req.query.limit) || 20;
  const skipIndex = (page - 1) * limit;
  
  // Ambil 2 query sejajar secara Promise Asinkron tanpa memblokade baris koding (Faster!)
  const [usersInfo, totalUserCount] = await Promise.all([
    prisma.user.findMany({ skip: skipIndex, take: limit }),
    prisma.user.count()
  ]);

  return res.json({
    success: true,
    data: usersInfo,
    meta: {
      totalItems: totalUserCount,
      totalPages: Math.ceil(totalUserCount / limit),
      currentPage: page,
      limitPerPage: limit
    }
  });
};

Injeksi Token JWT Otomatis (Client-Side Axios)

Setup File instance axios di React disetel agar secara global & konstan diinjeksikan header JWT sebelum menembak Endpoint. Mencegah Dev harus menulis logic Auth secara berulangulang manual

frontend/src/services/apiClient.js
import axios from 'axios';

const apiClient = axios.create({
  // Versioning URL: Pakai v1 untuk membedakan apabila nanti ada pembaruan arsitektur ke v2
  baseURL: 'https://api.aichat.com/v1',
  headers: {
    'Content-Type': 'application/json'
  }
});

// Interceptor: Menambahkan token sebelum API berangkat (Request Trigger)
apiClient.interceptors.request.use((config) => {
  const token = localStorage.getItem('auth_token');
  if (token) {
    config.headers['Authorization'] = `Bearer ${token}`; // Injeksi otomatis JWT
  }
  return config;
}, (error) => {
  return Promise.reject(error);
});

export default apiClient;

  • Endpoint agregasi kalkulasi Laba/Rugi, Overheads, dan Pajak total (Tampilan Dashboard Grafik harian/bulanan)
  • Analitik produk Best-Seller diurutkan via Query
Printer Driver [POST] /print/*
  • Bridge penerus instruksi (Buffer stream byte) ke alat cetak mesin Bluetooth/Thermal USB 58mm POS
  • Triggers pembuka mekanis (Cash Drawer pulse)
Realtime APIs [GET/POST] /chat/* & /weather/* & /reminder/*
  • Sistem kirim & muat riwayat modul Live Chat
  • Manajemen Reminder / Pengingat Pop-up alarm staff
  • Integrasi API pihak ketiga penarik forecast Cuaca (Suhu derajat & kelembapan) harian
Facility & Building [CRUD] /facility/* & /building/*
  • Inventarisasi manajemen pencatatan cabang / aset gedung operasional
  • Pencatatan fasilitas penunjang seperti listrik, internet (Wifi), dan perawatan CCTV toko

Catatan IT: Total modul mencapai 24 file `.route.ts`. Endpoint ini diwajibkan melewati proteksi Authentication Middleware dan validasi Request schema via middleware Joi terpusat.

9. Panduan Training AI (Self-Learning System)

Fitur AI Training dirancang agar Chatbot dapat belajar secara mandiri dari interaksi user yang tidak terjawab, sehingga akurasi jawaban meningkat seiring waktu tanpa perlu koding ulang

Alur Kerja Otomasi AI (Self-Learning)

graph TD A["User Bertanya"] --> B{"Cek Knowledge Base"} B -- "Ditemukan (High Score)" --> C["Kirim Jawaban FAQ"] B -- "Tidak Ditemukan / Skor Rendah" --> D["Masuk Ke Log Unresolved"] D --> E["Self-Learning Service (Cron)"] E --> F["Analisis Pertanyaan & Prediksi Jawaban"] F --> G["Generate Suggestion (AITraining Dashboard)"] G --> H{"Admin Review"} H -- "Approve" --> I["Tambah Otomatis ke Knowledge Base"] H -- "Reject" --> J["Hapus Saran"] I --> K["Sync ke Vector DB (ChromaDB)"] K --> L["AI Menjadi Lebih Pintar"]

Komponen Utama Control Center

  1. Knowledge Base: Database pengetahuan utama berisi FAQ (Tanya Jawab). Setiap data di sini akan dikonversi menjadi Vector Embedding agar AI paham konteks kalimat, bukan hanya kata kunci
  2. AI Training (Suggestions): Area di mana AI memberikan "saran" jawaban berdasarkan chat user yang gagal dijawab sebelumnya. Admin cukup melakukan verifikasi
  3. Intent Rules: Aturan spesifik untuk memicu aksi tertentu (misal: memanggil API cek stok atau melakukan handoff ke Admin manusia jika user marah)

Studi Kasus & Panduan Contoh Training

Berikut adalah simulasi bagaimana Admin melatih AI dalam skenario nyata Toko Retail/Bengkel:

Kasus 1: Produk Baru Belum Ada di Database

Masalah: Banyak pelanggan bertanya "Ada stok ban Pirelli ukuran 17?" tapi AI menjawab "Maaf, saya tidak tahu".

Langkah Solusi:
  1. Buka menu AI Training
  2. Temukan log pertanyaan tersebut. AI akan menyarankan jawaban (Suggestion) dari data produk yang ada atau input manual
  3. Klik Approve
  4. Hasil: Besoknya, jika ada yang tanya ban Pirelli, AI langsung jawab dengan stok terbaru

Kasus 2: Menangani "Bahasa Gaul" Pelanggan

Masalah: Pelanggan bertanya "Bengkel buka jam piro?" (bahasa daerah). AI awalnya bingung karena di database hanya ada "Jam operasional".

Langkah Solusi:
  1. Buka Knowledge Base, edit FAQ tentang Jam Operasional
  2. Tambahkan di kolom Alternatives: "buka jam piro", "jam buka", "closing jam berapa".
  3. Simpan. AI sekarang melakukan Semantic Mapping
  4. Hasil: AI tetap paham maksud user meskipun bahasanya tidak formal

💡 Tips Cara "Melatih" AI agar Cepat Pintar:

  • Gunakan FAQ yang Detail: Masukkan se-variatif mungkin pertanyaan yang bermakna sama di bagian Alternatives
  • Rajin Review Dashboard: Cek menu AI Training secara berkala. Berikan Approve pada saran yang relevan
  • Gunakan Kategori: Mengelompokkan FAQ dalam kategori membantu AI membatasi ruang pencarian informasi yang lebih akurat
