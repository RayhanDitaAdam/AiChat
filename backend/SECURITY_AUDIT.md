# Web Application Security Audit \u0026 Hardening Report\n\n## 1. Vulnerability Findings \u0026 Severity\n\n### \u003cspan style=\"color:#e74c3c\"\u003e[Critical]\u003c/span\u003e Persistent Cross-Site Scripting (XSS) via File Uploads\n**Location:** \u0060src/common/middleware/upload.middleware.ts\u0060\n**Attack Vector:** The application allowed file filtering based purely on the client-provided MIME type (e.g., \u0060mimetype.startsWith(\u0027image/\u0027)\u0060). An attacker could upload an HTML file painted with \u0060Content-Type: image/png\u0060. Since the backend respects the original file extension (\u0060path.extname(file.originalname)\u0060), it writes the file as \u0060.html\u0060. When victims navigate to \u0060/uploads/malicious.html\u0060, it executes JavaScript in the victim\u0027s browser context.\n**Mitigation:** Enforced strict File Extension checking against a curated whitelist of standard images, converting \u0060extname\u0060 to lowercase to prevent \u0060.pHp\u0060 bypasses, and checking both MIME and Extension dynamically.\n\n### \u003cspan style=\"color:#e74c3c\"\u003e[Critical]\u003c/span\u003e Tokens Stored Client-Side Disclosing Sessions to XSS\n**Location:** \u0060src/modules/auth/auth.controller.ts\u0060\n**Attack Vector:** Expressing authentication tokens via JSON payload instead of robustly protected \u0060HttpOnly\u0060, \u0060secure\u0060 cookies inherently implies that frontend libraries must hold these in volatile or local storage scopes. Any arbitrary XSS payload retrieved could simply exfiltrate localStorage secrets, causing session hijacks.\n**Mitigation:** We need to persist authorization via \u0060HttpOnly\u0060 cookies in the controller. I updated the \u0060auth.middleware.ts\u0060 logic strictly verifying tokens supplied via \u0060req.cookies.accessToken\u0060 alongside initial header verification. \n\n### \u003cspan style=\"color:#f39c12\"\u003e[High]\u003c/span\u003e Cross-Site Request Forgery (CSRF) Protection Disabled\n**Location:** \u0060src/app.ts\u0060\n**Attack Vector:** Double CSRF synchronization logic was commented out entirely: \u0060// CSRF disabled temporarily to unblock user\u0060. \n**Mitigation:** The CSRF module was re-enabled globally excluding whitelisted \u0060GET\u0060 contexts and specific integration endpoints. Reverted \u0060sameSite\u0060 rules back to \u0060\u0027strict\u0027\u0060.\n\n### \u003cspan style=\"color:#f39c12\"\u003e[High]\u003c/span\u003e Authentication Endpoints Lacking Granular Rate Limiting\n**Location:** \u0060src/modules/auth/auth.route.ts\u0060\n**Attack Vector:** While the Express app possessed a 1000/15min global rate limiter, the \u0060/login\u0060, \u0060/reset-password\u0060, and \u0060/2fa\u0060 routes didn\u0027t have explicit rate limitation layers. An attacker could rotate passwords leveraging automated tools at ~1 API hit per second.\n**Mitigation:** Bound a specific \u006010 requests per 15 mins\u0060 limit specifically to all sensitive Authentication mutating POST routes.\n\n### \u003cspan style=\"color:#f1c40f\"\u003e[Medium]\u003c/span\u003e CORS Over-Permissive Rules \u0026 DoS Payloads\n**Location:** \u0060src/app.ts\u0060\n**Attack Vector:** The regex parsed cross-origin allowed arbitrary requests originating from any local-network topology (e.g. \u0060192.168.0.x\u0060, \u006010.x.x.x\u0060). Malicious local tenancies hosted in the same cloud infra could bypass CORS restrictions. Additionally, Express was accepting payloads up to 10mb in body length.\n**Mitigation:** Limited maximum Request entity lengths strictly to \u00601mb\u0060. Deprecated the promiscuous Regex matching for explicit arrays of whitelisted origins.\n\n\n---\n\n## 2. Recommended Code-Level Improvements (Pending Refactor due to Breaking Frontend impact)\n\n### A. Migrating Auth Tokens to \u0060HttpOnly\u0060 Cookies\n\nThe app currently returns JSON payload with \u0060accessToken\u0060. Modify all controllers emitting tokens in \u0060auth.controller.ts\u0060:\n\n\u0060\u0060\u0060ts\n// Before:\nres.json({ status: \u0027success\u0027, accessToken, user });\n\n// After:\nres.cookie(\u0027accessToken\u0027, accessToken, {\n    httpOnly: true,\n    secure: process.env.NODE_ENV === \u0027production\u0027,\n    sameSite: \u0027strict\u0027,\n    maxAge: 60 * 60 * 1000 // 1 hour\n});\nres.cookie(\u0027refreshToken\u0027, refreshToken, { \n    httpOnly: true, secure: true, sameSite: \u0027strict\u0027, maxAge: 7 * 24 * 60 * 60 * 1000 \n});\nres.json({ status: \u0027success\u0027, user });\n\u0060\u0060\u0060\n\n*Note*: Updating this will require adding \u0060withCredentials: true\u0060 to frontend Axios/fetch API configurations.\n\n### B. Remove Uploads from Direct Application Server Root\nDo not serve uploads folder out of the Express core (\u0060app.use(\u0027/uploads\u0027, express.static)\u0060). \nInstead, pipe uploads directly to an S3 Object Storage bucket ensuring no filesystem execution context.\n\n---\n\n## 3. Secure Deployment Configuration\n\nFor production readiness spanning containerization to reverse proxies:\n\n### **Docker \u0026 Subsystems**\n1. **Non-Root Execution:** Node instances must never run as \u0060root\u0060 in containers. Define \u0060USER node\u0060 at the end of the Dockerfile.\n2. **Read-Only Root Filesystem:** Deploy containers using \u0060--read-only\u0060 configurations, allocating memory tmpfs to \u0060/tmp\u0060.\n\n### **Nginx Reverse Proxy Configurations**\nMask your Node ecosystem behind Cloudflare or Nginx, encrypting via TLS 1.3 exclusively.
